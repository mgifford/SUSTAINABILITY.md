Responsive Layout Patterns: Working Demo

This page demonstrates layout techniques required by WSG 3.6 — Ensure Layouts Work for Different Devices and Requirements. It is validated automatically by html-validate in CI.

1. Container queries instead of page queries

The card grid below adapts based on how much space its container has, not the viewport width. This makes the component reusable anywhere on the page regardless of surrounding layout.

  • Mobile-first defaults

    One column. No container query needed. All content is readable without CSS.

  • Tablet-width container

    At 36 rem inline size the grid switches to two columns using @container card-grid (min-inline-size: 36rem).

  • Desktop-width container

    At 60 rem the grid expands to three columns, still driven by the parent container rather than the viewport.

2. Container-aware sidebar

Article content and an aside stack vertically on narrow containers and shift to a two-column layout when there is room.

Main content area

This column takes two-thirds of the available space when the container is at least 48 rem wide.

On narrow containers it occupies the full width so text remains legible without horizontal scrolling.

Aside

Supplementary content. Stacks below main content on small containers; moves alongside it on wider ones.

3. Carbon-aware motion

Transitions are activated only when the user has not requested reduced motion. This avoids unnecessary GPU work on devices that prefer quieter rendering.

Hover the badge below. On systems with prefers-reduced-motion: reduce the badge is static.

Hover me (motion opt-in)

CSS pattern: 

@media (prefers-reduced-motion: no-preference) {
  .element { transition: transform 0.2s ease; }
}

4. Carbon-aware data loading

Decorative images and heavy backgrounds are skipped when prefers-reduced-data: reduce is set. All text content and layout remain fully functional.

Decorative illustration (loaded only when prefers-reduced-data is not set)

CSS pattern: 

@media (prefers-reduced-data: no-preference) {
  .hero { background-image: url("hero.webp"); }
}

5. Progressive enhancement checklist

Each layer adds capability without breaking the layer below.

  1. HTML — semantic, readable content with no CSS or JavaScript dependency.
  2. CSS baseline — mobile-first defaults; single column, readable typography, adequate touch targets.
  3. Container queries — components respond to their container, enabling reuse across sidebars, modals, and full-page layouts without duplication.
  4. Carbon-aware enhancements — transitions, decorative images, and animations are added only when the user has not opted out via OS or browser preferences.
